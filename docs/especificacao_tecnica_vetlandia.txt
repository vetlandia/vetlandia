3. AUTENTICAÇÃO E SEGURANÇA
--------------------------------------------------------------------------------
- Tokens:        JWT via python-jose 3.3.0
- Algoritmo:     HS256 (config: ALGORITHM)
- Expiração:     ACCESS_TOKEN_EXPIRE_MINUTES = 1440 (24 horas)
- Hash de senha: passlib[bcrypt] 1.7.4 + bcrypt (<5.0)  -> CryptContext bcrypt
- Transporte:    Cookie "access_token" httpOnly, samesite=lax
                 (logout em GET /logout apaga o cookie e redireciona para /)
- Assinatura:    SECRET_KEY (variável de ambiente)
- Implementação: app/core/security.py
                 (verify_password, get_password_hash,
                  create_access_token, decode_access_token)

FLUXO RESET DE SENHA ("Esqueci minha senha"):
  GET  /esqueci-senha           -> formulário (template auth/esqueci-senha.html)
  POST /api/auth/forgot-password -> gera secrets.token_urlsafe(32), armazena
                                   SHA-256 hash em password_reset_tokens, envia
                                   e-mail com link. Sempre retorna 200 (anti-enum.)
  GET  /redefinir-senha?token=  -> formulário de nova senha (auth/redefinir-senha.html)
  POST /api/auth/reset-password -> valida token (expira 1h), atualiza password_hash,
                                   marca token used=True

5. PAINEL ADMINISTRATIVO (/admin)
--------------------------------------------------------------------------------
Acesso: require_admin — user_type == ADMIN.
Prefixo: montado em "/admin" no main.py.

PÁGINAS:
  GET  /admin                   Dashboard principal (pendências, avaliações,
                                recomendações, entitlements, histórico)
  GET  /admin/analytics         Analytics com filtro de data (?date_from=&date_to=)
  GET  /admin/veterinarios      Lista todos os vets; ações inline
  GET  /admin/clinicas          Lista todas as clínicas; ações inline
  GET  /admin/tutores           Lista todos os tutores; ações inline

AÇÕES — VETERINÁRIOS:
  POST /admin/vets/{id}/approve   Aprovar (redireciona para redirect_to)
  POST /admin/vets/{id}/reject    Reprovar reversível (is_rejected=True) + e-mail
  POST /admin/vets/{id}/restore   Reverter reprovação (is_rejected=False)
  POST /admin/vets/{id}/delete    Excluir vet + conta de usuário (cascade completo)
  POST /admin/vets/{id}/set-badge Atribuir badge
  POST /admin/vets/{id}/verify    Marcar CRMV verificado
  GET  /admin/vets/{id}/carteira  Visualizar carteirinha (redirect para crmv_card_url)
  GET  /admin/vets/{id}/validar-crmv Tela de validação do CRMV

AÇÕES — CLÍNICAS:
  POST /admin/clinics/{id}/approve           Aprovar + e-mail
  POST /admin/clinics/{id}/reject            Reprovar reversível + e-mail
  POST /admin/clinics/{id}/restore           Reverter reprovação
  POST /admin/clinics/{id}/delete            Excluir clínica + conta de usuário
  POST /admin/clinics/{id}/set-badge         Atribuir badge
  POST /admin/clinics/{id}/verify            Marcar verificado
  POST /admin/clinics/{id}/recruitment-access Habilitar acesso à página de recrutamento
  POST /admin/clinics/{id}/entitlement       Toggle de produto/feature (entitlements)

AÇÕES — AVALIAÇÕES:
  POST /admin/reviews/{id}/approve  Publicar + e-mails (autor e avaliado)
  POST /admin/reviews/{id}/reject   Rejeitar + e-mails (autor e avaliado)
  POST /admin/reviews/{id}/delete   Excluir permanentemente

AÇÕES — RECOMENDAÇÕES:
  POST /admin/recommendations/{id}/approve  Publicar + e-mails
  POST /admin/recommendations/{id}/reject   Rejeitar + e-mails
  POST /admin/recommendations/{id}/delete   Excluir

AÇÕES — USUÁRIOS (genéricas, usadas pelas sub-páginas):
  POST /admin/users/{id}/block    Inativar conta (is_active=False) + desaprova perfil
  POST /admin/users/{id}/unblock  Reativar conta (is_active=True) + reativa perfil
  POST /admin/users/{id}/delete   Excluir conta + todos os dados vinculados (cascade)
  POST /admin/users/bulk-action   Ação em massa (approve/inativar/excluir)

PADRÃO redirect_to:
  Todas as rotas de ação aceitam redirect_to: str = Form("/admin").
  As sub-páginas passam redirect_to=/admin/veterinarios (ou /clinicas ou /tutores)
  para que o admin permaneça na sub-página após a ação.

EXCLUSÃO COMPLETA (_delete_user_and_profiles):
  Apaga em cascata: casos clínicos + comentários dos casos, avaliações recebidas
  + comentários dessas avaliações, avaliações escritas pelo usuário, perfil
  (vet/clinic/tutor), conta de usuário. Evita violações de FK.

12. FLUXO DE DEPLOY
--------------------------------------------------------------------------------
1. Editar código localmente.
2. git push origin main.
3. Railway detecta push → build NIXPACKS → roda "alembic upgrade head"
   → sobe Uvicorn em $PORT.
4. Cloudflare serve via CDN. HTML reflete imediatamente; CSS/JS dependem
   do cache (~4h) — o versionamento ?v= garante que o browser busca o novo.

ATENÇÃO: commit apenas LOCAL não atualiza o site. Só após git push em origin/main.

13. REGRAS DE NEGÓCIO RELEVANTES
--------------------------------------------------------------------------------
FAIXAS DE PREÇO (clínicas):
  consulta_preco_min/max, procedimento_preco_min/max — campos opcionais (Integer).
  Validação: @model_validator em ClinicBase e ClinicUpdate rejeita min > max
  com mensagem em PT-BR. Validação também no frontend (JS) antes do submit.
  Disponível no cadastro inicial e na edição de perfil (minha-conta.html).

EXCLUSÃO DE USUÁRIO:
  _delete_user_and_profiles() remove em cascata: comentários de casos,
  casos clínicos, avaliações recebidas e escritas (+ comentários),
  perfil (vet/clinic/tutor) e conta de usuário. Evita violações de FK.
  Usado em delete_vet, delete_clinic e delete_user.

REPROVAÇÃO REVERSÍVEL (Módulo 9):
  Vets e clínicas têm is_rejected (bool). Reprovar não apaga o cadastro —
  apenas marca is_rejected=True e is_approved=False. Admin pode restaurar
  (is_rejected=False), voltando para a fila de pendentes.

ENTITLEMENTS (Módulo 8):
  Produtos/funcionalidades liberadas por clínica via ClinicEntitlement.
  Toggle via POST /admin/clinics/{id}/entitlement.
  Produto atual: acesso à página de recrutamento.

RECRUTAMENTO:
  Página /recrutamento — visível apenas para clínicas com has_recruitment_access.
  Lista vets com disponibilidades de recrutamento (disp_plantao, disp_volante,
  disp_oportunidades, disp_parcerias, disp_estagio).
  Filtros: tipo (todos/formados/estudantes), disponibilidade, nome.

RESET DE SENHA:
  Token raw: secrets.token_urlsafe(32) → enviado no link por e-mail.
  Token armazenado: SHA-256 do raw (never store plaintext).
  Expiração: 1 hora. Token único por usuário (novo request invalida anterior).
  Anti-enumeração: POST /forgot-password sempre retorna 200.

ANALYTICS — LGPD:
  IPs armazenados apenas como SHA-256 (irreversível). Nunca o IP bruto.
  viewer_user_id registrado apenas quando usuário está logado.

CADASTRO DE VETERINÁRIO:
  crmv_card_url: URL da carteirinha CRMV (data URL ou https) — salva no Clinic
  durante o cadastro inicial e editável em minha-conta.
  Estudantes (is_student=True): crmv não obrigatório; crmv_card_url N/A.

SLUGS:
  Gerados via slugify(nome + cidade). Garantia de unicidade por sufixo numérico
  (-1, -2, ...) em loop até encontrar slug livre.

--------------------------------------------------------------------------------
14. PADRÕES DE CÓDIGO
--------------------------------------------------------------------------------
- Scripts inline nos templates devem ser AUTOSSUFICIENTES — main.js carrega no
  final do body. Não depender de funções de main.js em scripts de página.
- Casos clínicos ainda usam localStorage/isAuthenticated() — incompatível com
  cookie httpOnly. Corrigir ao retomar esse módulo.
- Fire-and-forget para e-mail e analytics: threading.Thread(daemon=True).start().
  Falhas logadas, nunca propagadas.
- Redirect_to pattern: rotas de ação admin aceitam redirect_to via Form para
  sub-páginas permanecerem ativas após a ação.
- Emails: todos via send_email() (async) exceto quando precisa garantir entrega
  antes de retornar (usar send_email_sync).
